# Nagi’s Heart — 产品需求文档 PRD v2.0

> 文档类型：产品设计 PRD  
> 产品类型：日乙视觉小说 / 互动剧情游戏  
> 主版本：移动端竖屏体验  
> 兼容版本：网页版独立实现  
> 文档范围：产品定位、核心玩法、页面结构、交互体验、路线系统、图鉴回想、存档与设置  
> 不包含：Android / Web 技术架构、具体代码实现、接口设计、工程目录、数据库方案

---

## 1. 产品概述

### 1.1 产品名称

**Nagi’s Heart**

正式产品文案中，产品名统一使用 **Nagi’s Heart**。

不建议使用：

- Blue Lock: Nagi’s Heart
- Nagi’s Heart 梦向乙游
- Ant’s Heart
- 凪诚士郎心动记录

以上名称可作为内部讨论或早期草案，不作为正式产品主标题。

---

### 1.2 产品类型

本产品是 **日乙类型的单男主视觉小说游戏**。

核心结构是：

```text
剧情阅读
→ 关键选择
→ 变量累积
→ 路线分歧
→ 结局达成
→ 图鉴 / 回想 / 二周目回收
```

本产品不是国乙养成游戏，不设计以下系统作为主玩法：

- 抽卡
- 卡牌养成
- 日程养成
- 战斗系统
- 体力系统
- 每日任务
- 限时活动
- 多男主卡池
- 换装养成
- 付费数值成长

---

### 1.3 核心体验定位

一句话定义：

```text
玩家通过自己的心，去撞开 Nagi 的心。
```

玩家不是单纯“攻略 Nagi”，也不是把 Nagi 养成世界第一。玩家通过凝视、靠近、理解、陪伴、推动、克制或失控，最终抵达不同的关系结局。

本产品的体验核心不是“男主爱我”，而是：

```text
我如何爱他。
我是否真正看见他。
我的爱是否让 Nagi 更成为 Nagi。
```

---

### 1.4 产品主轴

产品表层必须像乙女游戏：

- 心动
- 暧昧
- 低反应糖点
- LINE 聊天
- 开放日重逢
- 公寓同居
- 季节约会
- 亲密关系
- 轻喜剧日常
- 结局回收

产品底层承载：

- Nagi 的足球成长
- Nagi 从无所谓到拥有欲望
- 世界舞台、媒体、资本与被看见
- 玩家对 Nagi 的理想化与控制风险
- 爱、资源、归处与笼子的边界
- Dream / Stay / Bad 三种关系走向

表层感受必须优先。玩家首先进入的是恋爱，不是体育资本论文。

---

## 2. 平台与版本定位

### 2.1 主版本

主版本为：

```text
移动端竖屏视觉小说体验
```

设计优先级：

1. 手机竖屏阅读舒适度
2. 单手点击推进
3. 长剧情阅读节奏
4. 存读档与回收便利
5. 章节 / 路线 / 图鉴结构清晰

---

### 2.2 网页版定位

网页版为后续独立实现版本，不是主版本，但产品规则应与主版本一致。

网页版不应重新设计成横向网页游戏，而应优先采用：

```text
桌面浏览器中的移动端游戏画布
```

即：桌面端居中展示接近手机比例的视觉小说界面，保留移动端主体验。

---

### 2.3 Android 与 Web 的关系

Android 和 Web 可以是两套技术实现，但必须共享同一套产品规则：

- 同一剧情结构
- 同一节点逻辑
- 同一变量定义
- 同一路线判定
- 同一结局条件
- 同一命名规则
- 同一图鉴 / 回想 / 存档体验标准

本 PRD 不规定技术实现方式，只规定产品表现和体验要求。

## 3. 用户与使用场景

### 3.1 目标用户

核心用户：

- 喜欢 Nagi 的女性向用户
- 喜欢日乙视觉小说的玩家
- 喜欢强剧情、多结局、情绪分歧的玩家
- 接受长篇剧情阅读与二周目回收的玩家

非核心用户：

- 主要追求抽卡养成的国乙玩家
- 主要追求强数值成长的养成游戏玩家
- 不喜欢长文本阅读的用户
- 只希望看单一甜宠结局的用户

---

### 3.2 主要游玩场景

移动端主要场景：

- 睡前阅读
- 通勤碎片时间
- 章节式推进
- 回看喜欢的剧情
- 为达成不同结局从分歧点重玩

网页端主要场景：

- 桌面大屏回看剧情
- 整理路线 / 图鉴
- 较长时间沉浸阅读
- 后续开发调试或内容预览

---

## 4. 核心玩法循环

### 4.1 主循环

```text
进入游戏
→ 阅读剧情
→ 做出关键选择
→ 触发角色回应
→ 变量 / 旗标变化
→ 推进下一节点
→ 进入路线分歧
→ 达成结局
→ 解锁图鉴 / 回想 / 彩蛋
→ 从分歧点重玩
```

---

### 4.2 二周目循环

日乙用户天然会回收不同路线和结局，因此必须支持：

- 已读文本跳过
- 跳到下一个选项
- 从已解锁节点重开
- 关键分歧点回看
- 结局图鉴回收
- CG 图鉴回收
- 事件回想
- LINE 回想
- TRUE END 后彩蛋回看

产品不应惩罚玩家回收路线。

---

## 5. 命名与称呼规则

### 5.1 玩家名

玩家首次游玩时录入：

```text
玩家名字：{{playerName}}
```

玩家可见文本中，不固定使用 Ant 作为姓名。

正确口径：

- 官方文案：你 / 玩家 / {{playerName}}
- 正文对话：{{playerName}}
- 内部设计：可使用 Ant 作为默认代号

---

### 5.2 Nagi 称呼

本轮全量正式产品不提供“玩家自定义 Nagi 称呼”功能。

显示规则：

| 场景 | 显示 |
|---|---|
| 玩家直呼角色 | Nagi |
| 旁白 | Nagi |
| 系统文案 | Nagi |
| 标题 / 章节 / 结局卡 | Nagi |
| 媒体 / 新闻 / 解说 | Nagi |
| Nagi 自称或他人称呼 | Nagi |
| 一一称呼 | Nagi少爷 |

旧 script 中残留的 `{{nagiCall}}` 仅视为历史占位符，不得进入任何玩家可见界面。本轮不做输入、不做替换链路、不作为主流程能力交付。

---

### 5.3 一一称呼规则

一一称呼玩家：

```text
Ant大小姐
```

该称呼为一一专属固定称呼，不随 `{{playerName}}` 改变。其他角色仍按各自关系和剧本语境使用 `{{playerName}}` 或其他称呼。

一一称呼 Nagi：

```text
Nagi少爷
```

一一是玩家公寓的智能中枢，不是万能 AI，不可读心，不替代人物判断。

---

## 6. 剧情内容基线

### 6.1 主线结构

当前产品剧情采用八部主线 + 结局篇：

```text
开场白
第一部 · 原作前置篇：他还没看见你，你已经看见了他
第二部 · 关系确立篇：开放日，突然进入彼此生活
第三部 · 前期低谷篇：NEL淘汰、生日与Side-B重返
第四部 · 世界看见他篇：U-20世界杯与豪门招募
第五部 · 世界杯后的夏天：同居、热恋与夏窗签约
第六部 · 豪门新星篇：新的房间，新的规则
第七部 · 发布会后东京返日假期：她站在光里
第八部 · 世界中心篇：他的名字
结局篇 · TRUE / GOOD / NORMAL / BAD
```

---

### 6.2 第一部最新节点链路

第一部最新链路为：

```text
p1 → p2 → c1a → c1b → u20j
```

对应功能：

| 节点 | 标题 | 功能 |
|---|---|---|
| p1 | 作战室·初遇 | 玩家最早从屏幕中看见 Nagi 的光 |
| p2 | 投资的私心 | 建立投资判断与私人心动的混合 |
| c1a | 会议室初见 | 第一次单独见面，加 LINE |
| c1b | 不麻烦的人 | 夜聊、精神补位、Nagi 主动把混乱放到玩家这里 |
| u20j | U-20日本代表战·被日本看见 | 从“玩家先看见”推进到“日本看见” |

旧节点 `e_select2` 不再作为第一部独立主线节点。

---

### 6.3 路线结构

本产品是单男主多路线结局。

路线不是“选不同男主”，而是：

```text
玩家如何爱 Nagi。
玩家如何理解 Nagi。
玩家是否把 Nagi 推成自己的理想。
Nagi 是否仍然拥有自己的名字。
```

主路线：

| 路线 | 核心命题 | 结局 |
|---|---|---|
| Dream | 他成名，你见证 | TRUE / GOOD |
| Stay | 他暂未成名，你陪伴 | NORMAL |
| Bad | 他成名，你加冕 | BAD |

---

### 6.4 结局定义

#### TRUE END：「世界第一，与你」

核心：

```text
Nagi 成为世界第一，也真正拥有自己的名字。
玩家在场见证，但不抢走他的主语。
他完成自己的战斗后，主动回到玩家身边。
```

达成体验：

- 情感完整
- Nagi 的 EGO 成立
- 玩家学会见证
- 爱没有压过人
- 世界与私人关系同时成立

---

#### GOOD END：「那么完美，那么爱你」

核心：

```text
他们看起来很完美，也确实相爱。
但缺少灵魂深处最危险也最真实的互相读懂。
```

达成体验：

- 高甜
- 关系保留
- 外部结果漂亮
- 但 Nagi 的某些真实反应没有被完全接住

---

#### NORMAL END：「普通情侣」

核心：

```text
Nagi 暂时没有成为世界中心。
但玩家在球员通道外接住了还没成名的他。
```

达成体验：

- 不是失败线
- 不是退役线
- 不是 Nagi 暗掉
- 是没有抵达最高舞台，但保住了真实关系

---

#### BAD END：「好麻烦」

核心：

```text
Nagi 成名了。
但这次成名被玩家推成了“她亲手加冕他”的故事。
最终，他不再让她靠近自己的世界。
```

达成体验：

- 不是不爱
- 不是 Nagi 失败
- 不是玩家恶毒
- 是爱得太正确、太用力，理想压过了人

---

## 7. 产品信息架构

### 7.1 一级页面

产品一级页面包括：

| 页面 | 功能 |
|---|---|
| 启动 / 主页 | 进入游戏、继续游戏、章节、图鉴、设置 |
| 开场白 | 初次进入时展示产品基调 |
| 名字设置 | 设置玩家名与 Nagi 称呼 |
| 章节地图 | 查看篇章、节点、已读进度、路线进度 |
| 游戏主界面 | 阅读剧情、推进对话、做选择 |
| 存档页 | 自动存档、手动存档、读档、删除存档 |
| 回想页 | 已读事件、LINE、章节回看 |
| 图鉴页 | CG、结局、彩蛋收集 |
| 设置页 | 文本、音频、显示、跳读、自动播放等设置 |
| 结局页 | 展示结局卡、解锁内容、回到图鉴或重玩 |

---

### 7.2 主页功能

主页提供：

- 继续游戏
- 从头开始
- 章节地图
- 存档
- 图鉴
- 回想
- 设置
- 开场白回看

首次进入时，优先引导：

```text
开场白 → 名字设置 → 第一部 p1
```

非首次进入时，默认展示：

```text
继续游戏
```

---

### 7.3 章节地图

章节地图用于承载日乙路线回收。

基础能力：

- 展示八部主线
- 展示每部已解锁节点
- 展示当前进行位置
- 展示已完成节点
- 展示未解锁节点为 `???`
- 已读节点可回看
- 可从关键分歧点重新开始
- 回看默认不覆盖主存档
- 选择“从此处重新开始”时才生成新进度

章节地图不应一开始暴露所有隐藏路线细节。

---

## 8. 游戏主界面

### 8.1 基础布局

移动端主界面采用竖屏视觉小说布局。

基本层级：

```text
背景层
氛围遮罩层
角色 / CG / 特殊演出层
对话框层
选择层
HUD / 系统菜单层
```

---

### 8.2 HUD

正式玩家 HUD 应克制，不应常驻显示大量数值。

建议常驻：

- 返回 / 菜单
- 当前章节名
- 自动播放开关
- 跳读入口
- 存档入口

不建议常驻：

- 羁绊进度条
- 理解度
- 控制值
- 距离值
- 路线倾向
- 结局资格

原因：这是一款日乙视觉小说，不是数值养成游戏。玩家应该通过文本、反应、关系变化感知路线，而不是刷数值条。

---

### 8.3 对话显示

对话显示包含：

- 说话者名称
- 对话正文
- 逐字显示效果
- 点击推进提示
- 可跳过已读文本
- 可打开 backlog

角色名显示规则：

| 角色 | 显示 |
|---|---|
| 玩家 | {{playerName}} |
| Nagi | Nagi |
| 一一 | 一一 |
| 旁白 | 不显示角色名或显示为旁白样式 |
| 其他角色 | 使用正式角色名 |

---

### 8.4 旁白显示

旁白分为两种：

#### 底部旁白

用于短句、场景动作、心理过渡。

建议长度：

```text
45–70 个中文字符
上限约 80–90 个中文字符
```

#### 全屏旁白

用于连续叙事、情绪推进、场景转换、长心理段落。

建议长度：

```text
150–220 个中文字符
避免滚屏
```

连续三段以上旁白时，可自动合并为全屏旁白，但产品上应允许编辑指定显示方式。

---

### 8.5 LINE 模式

LINE 不是普通对话框换皮，而是特殊演出模块。

适用场景：

- 第一部夜聊
- Nagi 主动发消息
- 异地期间联系
- 关键比赛前后短信
- 回想中的聊天记录

LINE 模式应呈现：

- 手机聊天界面
- 玩家与 Nagi 的左右气泡
- 输入中状态
- 时间间隔感
- 已读 / 未读节奏
- 短句回复
- 可进入 LINE 回想

LINE 的功能不是单纯发糖，而是建立 Nagi 对玩家的精神补位依赖。

---

## 9. 选项系统

### 9.1 选项出现频率

本产品不追求高频互动。

选项应出现在：

- 完整事件后的关键反应点
- 关系推进点
- 路线分歧点
- Nagi 情绪需要被接住的节点
- 玩家是否见证 / 推动 / 控制的判断点

不应为了互动频率强行加选项。

---

### 9.2 选项数量

每次选项建议：

```text
2–3 个
```

特殊情况可为 1 个确认选项，但不建议频繁使用。

---

### 9.3 选项标签

玩家可见标签应保持日乙感，不宜过于系统化。

可见标签示例：

- 心动
- 靠近
- 理解
- 撒娇
- 认真
- 沉默
- 推他一把
- 留在这里
- 让他自己决定

内部标签可更细：

- affection
- understanding
- witness
- push
- comfort
- control
- avoid
- possess
- release

玩家不需要看到所有后台标签。

---

### 9.4 选择反馈

选择后应出现：

1. 玩家台词或动作
2. Nagi 的即时回应
3. 必要时插入旁白
4. 后续文本轻微差异
5. 隐性变量变化

不建议在正式界面显示：

```text
理解度 +3
控制值 +1
进入 Dream 线
Bad 风险上升
```

可用更沉浸的反馈替代：

- Nagi 停顿
- Nagi 回复变短
- Nagi 靠近
- Nagi 避开
- 一句特殊台词
- 背景音乐变化
- 短暂心跳演出

---

## 10. 变量与路线产品规则

### 10.1 变量可见性

正式玩家可见：

- 章节进度
- 已读 / 未读
- 结局收集
- CG 收集
- 回想收集

正式玩家不可见：

- 理解度
- 距离
- 控制倾向
- EGO
- 路线旗标
- TRUE / GOOD 资格条件

开发调试可见：

- 所有变量
- 当前节点
- 当前路线
- 当前结局资格
- 当前已读状态

---

### 10.2 核心变量方向

产品层建议收敛为以下核心维度：

| 维度 | 作用 |
|---|---|
| bond | 关系亲密度 |
| understanding | 是否读懂 Nagi |
| distance | Nagi 是否开始退开 |
| ego | Nagi 自主欲望是否成立 |
| control | 玩家是否过度安排 / 覆盖 Nagi |
| witness | 玩家是否在关键时刻见证但不抢主语 |
| independence | Nagi 是否拥有自己的名字与主语 |

变量不应服务于“刷满好感”，而应服务于不同关系质感。

---

## 11. 存档系统

### 11.1 自动存档

系统应在以下时机自动保存：

- 节点开始
- 节点结束
- 做出选择后
- 进入结局页
- 解锁图鉴内容后

自动存档保存：

- 当前节点
- 当前台词位置
- 玩家名
- Nagi 称呼
- 已读记录
- 变量状态
- 路线状态
- 图鉴解锁状态
- 结局解锁状态

---

### 11.2 手动存档

玩家可手动存档。

建议能力：

- 多档位
- 显示章节名
- 显示节点名
- 显示保存时间
- 显示路线阶段，不直接暴露隐藏路线
- 可覆盖
- 可删除
- 可读取

手动存档数量建议不少于 30 条。

---

### 11.3 快速存读档

日乙玩家需要快速试选项，因此应支持：

- 快速存档
- 快速读档
- 从当前选项前回退

### 17.1 产品体验验收

## 20. 2026-07-11 PM 强制更新：进入流程、存档、HUD 与剧本文本展示

> 本节为全量正式产品的强制产品规则。若与前文关于继续故事、存档、HUD、剧情回顾、旁白或章节过渡的旧描述冲突，以本节为准。

### 20.1 首次进入与后续进入

仅首次进入执行：

```text
开屏页 -> 开场白页 -> 名字设置页 -> 第一章第一页剧情页
```

后续进入执行：

```text
开屏页 -> 主页
```

规则：

- 只有存在默认退出进度时，主页才显示“继续故事”。
- 从未游玩时不显示“继续故事”。
- “继续故事”读取默认退出进度，不读取手动存档槽。
- “新的故事”重置默认退出进度，并从第一章第一页重新开始。
- “新的故事”默认不删除手动存档，除非玩家主动执行带二次确认的清除操作。
- 主页入口“存档”统一改为“存档进度”。

### 20.2 存档模型

产品必须区分两种进度：

- 默认进度：自动维护，专供“继续故事”读取。
- 手动存档：玩家点击剧情 HUD 的存档按钮产生，可在主页“存档进度”中查看。

HUD 存档必须：

- 保存当前节点、句子序号、变量、玩家名、路线状态和时间戳。
- 保存成功后明确提示“存档成功”。
- 在主页“存档进度”列表创建或更新对应记录。

### 20.3 剧情页 HUD

固定布局：

```text
左侧：返回
中间：当前章节标题
右侧：自动播放 / 停止、存档、剧情回顾
```

规则：

- 自动播放按钮在“自动播放”和“停止”之间切换。
- 存档按钮写入手动存档，并提示“存档成功”。
- 剧情回顾展示当前小章节从开头到当前句的全部已读剧情。
- 剧情 HUD 不再保留其他菜单，最右侧固定为“剧情回顾”。
- 剧情页必须提供“跳过本节”入口，但它不并入顶部 HUD 五键结构；作为剧情层的次级操作显示。
- “跳过本节”点击后弹出确认，确认后直接进入当前小章节结束页，不逐句播放剩余内容。
- 跳过行为需要解锁下一小章节，但不把未实际展示的文本写入剧情回顾。

### 20.4 人物对白与旁白

#### 20.4.1 基础规则

- 代码不得自动给人物对白添加引号。
- script 文本就是最终展示文案；确实需要引号时，由 script 自身写入。
- 同一场景、同一时间、同一叙事节拍下连续出现的无说话人旁白，合并为一个连续叙事块。
- 遇到人物对白、玩家选项、场景或时间切换、章节边界、脚本指定停顿时，结束当前叙事块。

#### 20.4.2 长短旁白判定

- 连续 1–2 段，且在底部旁白框中实际排版不超过 4 行：短旁白。
- 连续 3 段及以上：长旁白。
- 不足 3 段，但实际排版超过底部旁白框 4 行：长旁白。
- 单段约 90 个中文全角字符时触发编辑预警，但不作为运行时硬切分线。
- 长心理、重要情绪铺陈和说明性叙事可由编辑强制指定为长旁白，不受段数限制。

#### 20.4.3 展示区别

- 人物对白：底部对白框，显示人物名，使用对白正体字体。
- 短旁白：底部旁白框，不显示人物名，也不显示“旁白”标签；使用独立旁白字形。
- 长旁白：整页毛玻璃阅读层，不显示人物名或“旁白”标签；正文左对齐，按页展示。

#### 20.4.4 长旁白分页

- 长旁白禁止滚动，必须按当前设备和字体缩放下的实际排版高度分页。
- 编辑预估目标为每页约 180–240 个中文全角字符，但该范围不是运行时硬上限。
- 优先保留完整段落；单段超过一页时，在最接近页尾且不溢出的 `。！？……；` 等句末标点处分割。
- 避免标题、单行孤段或不足两行的段尾被单独留在下一页。
- 每页作为一个完整阅读单位一次性显示，不按段要求玩家连续点击。
- 非末页底部显示翻页图标和“当前页 / 总页数”；末页显示继续剧情引导。
- 自动播放必须逐页推进，并按本页有效字数计算停留时间，设置最短和最长阅读时长。

#### 20.4.5 script 数据要求

script 数据必须能够结构化表达：

- 人物对白、短旁白、长旁白三种语义显示类型；
- 连续叙事块 ID；
- 编辑强制合并与强制断开；
- 段落边界；
- 可选的人工分页提示；
- 章节开始页和结束页的结构化字段。

人工分页只用于保护叙事节奏，运行时仍须以设备实际排版结果为准。

#### 20.4.6 待 Ant 确认的视觉基线

- 人物对白：无衬线 Medium，18sp / 29sp，最多 4 行。
- 短旁白：衬线 Italic，16sp / 28sp，最多 4 行。
- 长旁白：衬线 Regular，18sp / 32sp，段间距 20dp，水平安全边距 32dp。
- 长旁白背景：初始建议 20dp 模糊并叠加约 48% 深色遮罩，须使用最亮和最暗剧情图复核对比度。
- 翻页提示：底部居中显示翻页图标及当前页 / 总页数，末页切换为继续引导。

### 20.5 选项与自动跳转

- `->` 不得作为玩家可见选项出现。
- 单一自动跳转属于引擎内部行为，不得渲染成选项卡。
- 玩家可见选项必须是有意义的选择或明确写作的回应。
- `{{playerName}}` 必须在正文、说话人、选项、回应、剧情回顾和存档摘要中完成替换，不得裸露。
- 历史 script 中残留的 `{{nagiCall}}` 本轮不得以裸文本出现在任何玩家可见位置；发现即视为 script 清理缺陷。

### 20.6 章节层级、开始页与结束页

产品采用两层章节结构：

- 大章：篇 / 部级章节，承担大的叙事阶段。
- 小章节：幕 / 节级章节，承担实际阅读推进单元。

强制规则：

- 大章和小章节都必须拥有各自独立的开始页与结束页，三类页面均隐藏剧情 HUD。
- 大章开始页用于宣告阶段进入；小章节开始页用于宣告当前具体剧情单元开始。
- 大章结束页用于阶段收束；小章节结束页用于本节阅读完成、跳转下一节或进入本章目录。
- 不允许只给大章做开始页而不给结束页；也不允许小章节直接从正文硬切入、硬切出。

大章开始页必显：

- 大章序号；
- 大章正式标题；
- “轻触开始 / 轻触继续”提示。

大章结束页必显：

- 完成状态；
- 大章序号；
- 大章正式标题；
- 中文完成提示；
- 有效主操作。

小章节开始页必显：

- 所属大章标识；
- 小章节序号；
- 小章节正式标题；
- “轻触开始 / 轻触继续”提示。

小章节结束页必显：

- 完成状态；
- 小章节序号与标题；
- 下一步主操作；
- 可选次操作“本章目录”。

统一规则：

- 副标题、时间地点、情绪引句和解锁摘要仅在有实际内容时显示，不保留空占位。
- 有后续小章节时，小章节结束页主操作为“下一节”；没有后续小章节但仍有后续大章时，大章结束页主操作为“下一章”；全部主线结束后主操作为“返回主页”或进入结局相关流程。
- 进入任一结束页时更新默认进度；进入下一页后，默认进度定位到下一开始页或下一正文起点。
- 结束页自动记录默认进度，不属于手动存档，不显示“存档成功”。
- “跳过本节”落点必须是当前小章节结束页，而不是直接跳过整章或整条路线。
- 开始页和结束页是全量产品必备流程页，不是可选视觉包装。

### 20.7 剧情回顾、章节目录与章节回看

必须区分三种能力：

1. 剧情回顾：剧情页内即时查看当前小章节从开头到当前句的全部已读内容。
2. 章节目录：展示当前大章下全部小章节结构、顺序与解锁状态。
3. 章节回看：从章节目录进入，回看已经完成过的小章节整段内容。

剧情回顾规则：

- 只展示当前小章节内已实际显示过的内容。
- 包含对白、短旁白、长旁白分页后的已读页、已做出的玩家选择及其已显示回应。
- 不展示未来内容，不补入被“跳过本节”略过的未读文本。

章节目录规则：

- 至少按当前大章列出全部小章节。
- 每项必须展示小章节序号、标题和状态。
- 状态分为：未解锁、进行中、已完成、已跳过完成。
- 未解锁项不可点击；进行中项进入当前进度；已完成与已跳过完成项可进入章节回看。

章节回看规则：

- 仅允许查看已完成或已跳过完成的小章节。
- 默认不覆盖当前默认进度，不写入手动存档。
- 回看结束后返回章节目录或原入口页。

### 20.8 结局与章节收束关系

- 小章节结束页不等于结局页，大章结束页也不等于结局页。
- 结局必须通过独立结局流程页展示，不得用普通章节结束页冒充。
- 结局页需承接结局达成、结局卡展示、图鉴解锁和后续回看入口。
- 章节目录只管理章节推进；结局图鉴单独管理 TRUE / GOOD / NORMAL / BAD 等结局回收。

### 20.9 验收清单

- 全新安装：开屏页 -> 开场白页 -> 名字设置页 -> 第一章第一页剧情页。
- 有游玩记录后再次启动：开屏页 -> 主页。
- 无默认进度时不显示“继续故事”。
- “继续故事”精确恢复默认进度中的节点、句子和变量。
- “新的故事”重置默认进度并从第一章第一页开始，不删除手动存档。
- HUD 显示返回、章节标题、自动播放 / 停止、存档、剧情回顾，不显示其他菜单。
- 剧情页存在“跳过本节”入口，确认后进入当前小章节结束页。
- HUD 存档后提示“存档成功”，并在主页“存档进度”中出现记录。
- 剧情回顾显示当前小章节从开头到当前句的全部已读内容。
- 人物对白没有代码自动添加的引号。
- 连续长旁白使用整页毛玻璃和分页阅读，不滚动、不逐段点击。
- 不出现玩家可见的 `->` 选项。
- 不裸露 `{{playerName}}` 或历史 `{{nagiCall}}` 占位符。
- 每个大章和每个小章节均有开始页和结束页。

## 21. 2026-07-19 实机反馈产品补丁：主流程优先级、章节节奏与结局闭环

> Status: current authority override.
> Source: Ant 大小姐 2026-07-19 Android 实机反馈。
> 本节与前文冲突时，以本节为准。

### 21.1 主流程逻辑优先级

以下问题按 P0 主流程缺陷处理，优先于视觉细节：

1. 结局在非预期章节突然出现；
2. 结局展示后继续接回普通剧情；
3. 后续章节卡住、无法继续、无法返回；
4. 达成结局后回忆画廊未解锁对应结局。

开发不得只从 UI 层修补这些问题，必须审计剧情路由、story-data、状态机、进度持久化和画廊读取逻辑。

### 21.2 剧情回顾产品规则

- 剧情回顾打开默认显示第一页，即本小章节最早已读内容。
- 不再默认显示最后一页。
- 内容必须按移动端安全区完整分页，不得裁切底部文字。
- 分页容量由实际可显示高度决定，不强制固定 8 条。
- 移动端使用左右滑动换页，保留页码，不显示上一页 / 下一页文字按钮。

### 21.3 小章节结束页取消

- 小章节独立结束页从当前产品范围移除。
- 正常流程：小章节正文结束后，直接进入下一小章节开始页。
- 跳过本节：确认后直接进入下一小章节开始页；若没有下一小章节，则进入大章节结束页；若主线已收束，则进入结局流程。
- 大章节结束页仍保留。
- 旧文档中“小章节必须有结束页 / 跳过本节必须进入当前小章节结束页”的规则废止。

### 21.4 结局页与画廊闭环

- 结局页是独立终局页，不得由普通章节结束页替代。
- 结局页 UI 当前缺失，必须由 UI 设计补齐并经 Ant 确认后开发。
- 结局达成时必须立即解锁画廊对应结局，并持久化。
- 结局页展示后不得自动进入后续剧情正文。
- 结局页至少需要以下产品动作：返回主页、进入回忆画廊、重看本结局 / 相关章节入口。最终按钮命名和布局由 UI authority 确认。

### 21.5 文字可读性与长旁白宽度

- 所有剧情页压图文字必须保证可读性优先；当背景亮、复杂或高对比时，文字托底需要增强。
- 长旁白正文宽度应与底部单行旁白正文宽度同基准，不得明显更窄。
- 长旁白 / 剧情回顾 / 开始页 / 结束页 / 弹窗 / HUD 的可读性 token 由 UI authority 统一补齐，开发不得自行凭感觉调风格。
