# Nagi’s Heart — 产品需求文档 PRD v2.0

> 文档类型：产品设计 PRD  
> 产品类型：日乙视觉小说 / 互动剧情游戏  
> 主版本：移动端竖屏体验  
> 兼容版本：网页版独立实现  
> 文档范围：产品定位、核心玩法、页面结构、交互体验、路线系统、图鉴回想、存档与设置  
> 不包含：Android / Web 技术架构、具体代码实现、接口设计、工程目录、数据库方案

---

## 1. 产品概述

### 1.1 产品名称

**Nagi’s Heart**

正式产品文案中，产品名统一使用 **Nagi’s Heart**。

不建议使用：

- Blue Lock: Nagi’s Heart
- Nagi’s Heart 梦向乙游
- Ant’s Heart
- 凪诚士郎心动记录

以上名称可作为内部讨论或早期草案，不作为正式产品主标题。

---

### 1.2 产品类型

本产品是 **日乙类型的单男主视觉小说游戏**。

核心结构是：

```text
剧情阅读
→ 关键选择
→ 变量累积
→ 路线分歧
→ 结局达成
→ 图鉴 / 回想 / 二周目回收
```

本产品不是国乙养成游戏，不设计以下系统作为主玩法：

- 抽卡
- 卡牌养成
- 日程养成
- 战斗系统
- 体力系统
- 每日任务
- 限时活动
- 多男主卡池
- 换装养成
- 付费数值成长

---

### 1.3 核心体验定位

一句话定义：

```text
玩家通过自己的心，去撞开 Nagi 的心。
```

玩家不是单纯“攻略 Nagi”，也不是把 Nagi 养成世界第一。玩家通过凝视、靠近、理解、陪伴、推动、克制或失控，最终抵达不同的关系结局。

本产品的体验核心不是“男主爱我”，而是：

```text
我如何爱他。
我是否真正看见他。
我的爱是否让 Nagi 更成为 Nagi。
```

---

### 1.4 产品主轴

产品表层必须像乙女游戏：

- 心动
- 暧昧
- 低反应糖点
- LINE 聊天
- 开放日重逢
- 公寓同居
- 季节约会
- 亲密关系
- 轻喜剧日常
- 结局回收

产品底层承载：

- Nagi 的足球成长
- Nagi 从无所谓到拥有欲望
- 世界舞台、媒体、资本与被看见
- 玩家对 Nagi 的理想化与控制风险
- 爱、资源、归处与笼子的边界
- Dream / Stay / Bad 三种关系走向

表层感受必须优先。玩家首先进入的是恋爱，不是体育资本论文。

---

## 2. 平台与版本定位

### 2.1 主版本

主版本为：

```text
移动端竖屏视觉小说体验
```

设计优先级：

1. 手机竖屏阅读舒适度
2. 单手点击推进
3. 长剧情阅读节奏
4. 存读档与回收便利
5. 章节 / 路线 / 图鉴结构清晰

---

### 2.2 网页版定位

网页版为后续独立实现版本，不是主版本，但产品规则应与主版本一致。

网页版不应重新设计成横向网页游戏，而应优先采用：

```text
桌面浏览器中的移动端游戏画布
```

即：桌面端居中展示接近手机比例的视觉小说界面，保留移动端主体验。

---

### 2.3 Android 与 Web 的关系

Android 和 Web 可以是两套技术实现，但必须共享同一套产品规则：

- 同一剧情结构
- 同一节点逻辑
- 同一变量定义
- 同一路线判定
- 同一结局条件
- 同一命名规则
- 同一图鉴 / 回想 / 存档体验标准

本 PRD 不规定技术实现方式，只规定产品表现和体验要求。

## 3. 用户与使用场景

### 3.1 目标用户

核心用户：

- 喜欢 Nagi 的女性向用户
- 喜欢日乙视觉小说的玩家
- 喜欢强剧情、多结局、情绪分歧的玩家
- 接受长篇剧情阅读与二周目回收的玩家

非核心用户：

- 主要追求抽卡养成的国乙玩家
- 主要追求强数值成长的养成游戏玩家
- 不喜欢长文本阅读的用户
- 只希望看单一甜宠结局的用户

---

### 3.2 主要游玩场景

移动端主要场景：

- 睡前阅读
- 通勤碎片时间
- 章节式推进
- 回看喜欢的剧情
- 为达成不同结局从分歧点重玩

网页端主要场景：

- 桌面大屏回看剧情
- 整理路线 / 图鉴
- 较长时间沉浸阅读
- 后续开发调试或内容预览

---

## 4. 核心玩法循环

### 4.1 主循环

```text
进入游戏
→ 阅读剧情
→ 做出关键选择
→ 触发角色回应
→ 变量 / 旗标变化
→ 推进下一节点
→ 进入路线分歧
→ 达成结局
→ 解锁图鉴 / 回想 / 彩蛋
→ 从分歧点重玩
```

---

### 4.2 二周目循环

日乙用户天然会回收不同路线和结局，因此必须支持：

- 已读文本跳过
- 跳到下一个选项
- 从已解锁节点重开
- 关键分歧点回看
- 结局图鉴回收
- CG 图鉴回收
- 事件回想
- LINE 回想
- TRUE END 后彩蛋回看

产品不应惩罚玩家回收路线。

---

## 5. 命名与称呼规则

### 5.1 玩家名

玩家首次游玩时录入：

```text
玩家名字：{{playerName}}
```

玩家可见文本中，不固定使用 Ant 作为姓名。

正确口径：

- 官方文案：你 / 玩家 / {{playerName}}
- 正文对话：{{playerName}}
- 内部设计：可使用 Ant 作为默认代号

---

### 5.2 Nagi 称呼

玩家首次游玩时录入：

```text
玩家对 Nagi 的称呼：{{nagiCall}}
```

显示规则：

| 场景 | 显示 |
|---|---|
| 玩家直接叫他 | {{nagiCall}} |
| 旁白 | Nagi |
| 系统文案 | Nagi |
| 标题 / 章节 / 结局卡 | Nagi |
| 媒体 / 新闻 / 解说 | Nagi |
| Nagi 自称或他人称呼 | Nagi |
| 一一称呼 | Nagi少爷 |

禁止全局把所有 `Nagi` 替换成 `{{nagiCall}}`。

---

### 5.3 一一称呼规则

一一称呼玩家：

```text
{{playerName}}小姐
```

一一称呼 Nagi：

```text
Nagi少爷
```

一一是玩家公寓的智能中枢，不是万能 AI，不可读心，不替代人物判断。

---

## 6. 剧情内容基线

### 6.1 主线结构

当前产品剧情采用八部主线 + 结局篇：

```text
开场白
第一部 · 原作前置篇：他还没看见你，你已经看见了他
第二部 · 关系确立篇：开放日，突然进入彼此生活
第三部 · 前期低谷篇：NEL淘汰、生日与Side-B重返
第四部 · 世界看见他篇：U-20世界杯与豪门招募
第五部 · 世界杯后的夏天：同居、热恋与夏窗签约
第六部 · 豪门新星篇：新的房间，新的规则
第七部 · 发布会后东京返日假期：她站在光里
第八部 · 世界中心篇：他的名字
结局篇 · TRUE / GOOD / NORMAL / BAD
```

---

### 6.2 第一部最新节点链路

第一部最新链路为：

```text
p1 → p2 → c1a → c1b → u20j
```

对应功能：

| 节点 | 标题 | 功能 |
|---|---|---|
| p1 | 作战室·初遇 | 玩家最早从屏幕中看见 Nagi 的光 |
| p2 | 投资的私心 | 建立投资判断与私人心动的混合 |
| c1a | 会议室初见 | 第一次单独见面，加 LINE |
| c1b | 不麻烦的人 | 夜聊、精神补位、Nagi 主动把混乱放到玩家这里 |
| u20j | U-20日本代表战·被日本看见 | 从“玩家先看见”推进到“日本看见” |

旧节点 `e_select2` 不再作为第一部独立主线节点。

---

### 6.3 路线结构

本产品是单男主多路线结局。

路线不是“选不同男主”，而是：

```text
玩家如何爱 Nagi。
玩家如何理解 Nagi。
玩家是否把 Nagi 推成自己的理想。
Nagi 是否仍然拥有自己的名字。
```

主路线：

| 路线 | 核心命题 | 结局 |
|---|---|---|
| Dream | 他成名，你见证 | TRUE / GOOD |
| Stay | 他暂未成名，你陪伴 | NORMAL |
| Bad | 他成名，你加冕 | BAD |

---

### 6.4 结局定义

#### TRUE END：「世界第一，与你」

核心：

```text
Nagi 成为世界第一，也真正拥有自己的名字。
玩家在场见证，但不抢走他的主语。
他完成自己的战斗后，主动回到玩家身边。
```

达成体验：

- 情感完整
- Nagi 的 EGO 成立
- 玩家学会见证
- 爱没有压过人
- 世界与私人关系同时成立

---

#### GOOD END：「那么完美，那么爱你」

核心：

```text
他们看起来很完美，也确实相爱。
但缺少灵魂深处最危险也最真实的互相读懂。
```

达成体验：

- 高甜
- 关系保留
- 外部结果漂亮
- 但 Nagi 的某些真实反应没有被完全接住

---

#### NORMAL END：「普通情侣」

核心：

```text
Nagi 暂时没有成为世界中心。
但玩家在球员通道外接住了还没成名的他。
```

达成体验：

- 不是失败线
- 不是退役线
- 不是 Nagi 暗掉
- 是没有抵达最高舞台，但保住了真实关系

---

#### BAD END：「好麻烦」

核心：

```text
Nagi 成名了。
但这次成名被玩家推成了“她亲手加冕他”的故事。
最终，他不再让她靠近自己的世界。
```

达成体验：

- 不是不爱
- 不是 Nagi 失败
- 不是玩家恶毒
- 是爱得太正确、太用力，理想压过了人

---

## 7. 产品信息架构

### 7.1 一级页面

产品一级页面包括：

| 页面 | 功能 |
|---|---|
| 启动 / 主页 | 进入游戏、继续游戏、章节、图鉴、设置 |
| 开场白 | 初次进入时展示产品基调 |
| 名字设置 | 设置玩家名与 Nagi 称呼 |
| 章节地图 | 查看篇章、节点、已读进度、路线进度 |
| 游戏主界面 | 阅读剧情、推进对话、做选择 |
| 存档页 | 自动存档、手动存档、读档、删除存档 |
| 回想页 | 已读事件、LINE、章节回看 |
| 图鉴页 | CG、结局、彩蛋收集 |
| 设置页 | 文本、音频、显示、跳读、自动播放等设置 |
| 结局页 | 展示结局卡、解锁内容、回到图鉴或重玩 |

---

### 7.2 主页功能

主页提供：

- 继续游戏
- 从头开始
- 章节地图
- 存档
- 图鉴
- 回想
- 设置
- 开场白回看

首次进入时，优先引导：

```text
开场白 → 名字设置 → 第一部 p1
```

非首次进入时，默认展示：

```text
继续游戏
```

---

### 7.3 章节地图

章节地图用于承载日乙路线回收。

基础能力：

- 展示八部主线
- 展示每部已解锁节点
- 展示当前进行位置
- 展示已完成节点
- 展示未解锁节点为 `???`
- 已读节点可回看
- 可从关键分歧点重新开始
- 回看默认不覆盖主存档
- 选择“从此处重新开始”时才生成新进度

章节地图不应一开始暴露所有隐藏路线细节。

---

## 8. 游戏主界面

### 8.1 基础布局

移动端主界面采用竖屏视觉小说布局。

基本层级：

```text
背景层
氛围遮罩层
角色 / CG / 特殊演出层
对话框层
选择层
HUD / 系统菜单层
```

---

### 8.2 HUD

正式玩家 HUD 应克制，不应常驻显示大量数值。

建议常驻：

- 返回 / 菜单
- 当前章节名
- 自动播放开关
- 跳读入口
- 存档入口

不建议常驻：

- 羁绊进度条
- 理解度
- 控制值
- 距离值
- 路线倾向
- 结局资格

原因：这是一款日乙视觉小说，不是数值养成游戏。玩家应该通过文本、反应、关系变化感知路线，而不是刷数值条。

---

### 8.3 对话显示

对话显示包含：

- 说话者名称
- 对话正文
- 逐字显示效果
- 点击推进提示
- 可跳过已读文本
- 可打开 backlog

角色名显示规则：

| 角色 | 显示 |
|---|---|
| 玩家 | {{playerName}} |
| Nagi | Nagi |
| 一一 | 一一 |
| 旁白 | 不显示角色名或显示为旁白样式 |
| 其他角色 | 使用正式角色名 |

---

### 8.4 旁白显示

旁白分为两种：

#### 底部旁白

用于短句、场景动作、心理过渡。

建议长度：

```text
45–70 个中文字符
上限约 80–90 个中文字符
```

#### 全屏旁白

用于连续叙事、情绪推进、场景转换、长心理段落。

建议长度：

```text
150–220 个中文字符
避免滚屏
```

连续三段以上旁白时，可自动合并为全屏旁白，但产品上应允许编辑指定显示方式。

---

### 8.5 LINE 模式

LINE 不是普通对话框换皮，而是特殊演出模块。

适用场景：

- 第一部夜聊
- Nagi 主动发消息
- 异地期间联系
- 关键比赛前后短信
- 回想中的聊天记录

LINE 模式应呈现：

- 手机聊天界面
- 玩家与 Nagi 的左右气泡
- 输入中状态
- 时间间隔感
- 已读 / 未读节奏
- 短句回复
- 可进入 LINE 回想

LINE 的功能不是单纯发糖，而是建立 Nagi 对玩家的精神补位依赖。

---

## 9. 选项系统

### 9.1 选项出现频率

本产品不追求高频互动。

选项应出现在：

- 完整事件后的关键反应点
- 关系推进点
- 路线分歧点
- Nagi 情绪需要被接住的节点
- 玩家是否见证 / 推动 / 控制的判断点

不应为了互动频率强行加选项。

---

### 9.2 选项数量

每次选项建议：

```text
2–3 个
```

特殊情况可为 1 个确认选项，但不建议频繁使用。

---

### 9.3 选项标签

玩家可见标签应保持日乙感，不宜过于系统化。

可见标签示例：

- 心动
- 靠近
- 理解
- 撒娇
- 认真
- 沉默
- 推他一把
- 留在这里
- 让他自己决定

内部标签可更细：

- affection
- understanding
- witness
- push
- comfort
- control
- avoid
- possess
- release

玩家不需要看到所有后台标签。

---

### 9.4 选择反馈

选择后应出现：

1. 玩家台词或动作
2. Nagi 的即时回应
3. 必要时插入旁白
4. 后续文本轻微差异
5. 隐性变量变化

不建议在正式界面显示：

```text
理解度 +3
控制值 +1
进入 Dream 线
Bad 风险上升
```

可用更沉浸的反馈替代：

- Nagi 停顿
- Nagi 回复变短
- Nagi 靠近
- Nagi 避开
- 一句特殊台词
- 背景音乐变化
- 短暂心跳演出

---

## 10. 变量与路线产品规则

### 10.1 变量可见性

正式玩家可见：

- 章节进度
- 已读 / 未读
- 结局收集
- CG 收集
- 回想收集

正式玩家不可见：

- 理解度
- 距离
- 控制倾向
- EGO
- 路线旗标
- TRUE / GOOD 资格条件

开发调试可见：

- 所有变量
- 当前节点
- 当前路线
- 当前结局资格
- 当前已读状态

---

### 10.2 核心变量方向

产品层建议收敛为以下核心维度：

| 维度 | 作用 |
|---|---|
| bond | 关系亲密度 |
| understanding | 是否读懂 Nagi |
| distance | Nagi 是否开始退开 |
| ego | Nagi 自主欲望是否成立 |
| control | 玩家是否过度安排 / 覆盖 Nagi |
| witness | 玩家是否在关键时刻见证但不抢主语 |
| independence | Nagi 是否拥有自己的名字与主语 |

变量不应服务于“刷满好感”，而应服务于不同关系质感。

---

## 11. 存档系统

### 11.1 自动存档

系统应在以下时机自动保存：

- 节点开始
- 节点结束
- 做出选择后
- 进入结局页
- 解锁图鉴内容后

自动存档保存：

- 当前节点
- 当前台词位置
- 玩家名
- Nagi 称呼
- 已读记录
- 变量状态
- 路线状态
- 图鉴解锁状态
- 结局解锁状态

---

### 11.2 手动存档

玩家可手动存档。

建议能力：

- 多档位
- 显示章节名
- 显示节点名
- 显示保存时间
- 显示路线阶段，不直接暴露隐藏路线
- 可覆盖
- 可删除
- 可读取

手动存档数量建议不少于 30 条。

---

### 11.3 快速存读档

日乙玩家需要快速试选项，因此应支持：

- 快速存档
- 快速读档
- 从当前选项前回退
- 从最近分歧点重开

---

## 12. Backlog / 跳读 / 自动播放

### 12.1 Backlog

Backlog 展示当前章节已播放文本。

功能：

- 查看历史台词
- 查看说话者
- 查看玩家选择
- 可回看最近对话
- 不改变当前游戏进度

Backlog 需要兼容：

- VN 对话
- 旁白
- LINE 消息
- 选项结果

---

### 12.2 已读跳过

已读跳过只跳过玩家已经看过的文本。

规则：

- 未读文本自动停止跳过
- 新选项处停止
- 新 CG / 特殊演出处停止
- 关键结局文本不应被误跳过

---

### 12.3 跳到下一个选项

提供“跳到下一个选项”功能。

适用场景：

- 二周目回收
- 从存档点试不同选择
- 快速进入分歧点

规则：

- 仅跳过已读文本
- 遇到未读文本停止
- 遇到新演出停止

---

### 12.4 自动播放

支持自动播放文本。

设置项：

- 自动播放开关
- 文字速度
- 句间停顿
- 是否在选项处暂停

---

## 13. 图鉴与回想

### 13.1 图鉴结构

图鉴分为：

```text
CG 图鉴
结局图鉴
事件回想
LINE 回想
特殊彩蛋
```

---

### 13.2 CG 图鉴

CG 图鉴用于收集剧情关键画面。

可按篇章分类：

- 第一部：作战室、会议室、U-20战
- 第二部：开放日、柠檬茶、公寓、棒棒糖
- 第三部：咖喱、生日、拥抱、亲密
- 第四部：世界杯、花环、豪门来信
- 第五部：同居、七夕、夏日祭、签约
- 第六部：曼城、新房间、采访、商业活动
- 第七部：东京返日、围巾、醉酒、打扮
- 第八部：关键战、通道、加冕、终局
- 结局：TRUE / GOOD / NORMAL / BAD
- 彩蛋：求婚

---

### 13.3 结局图鉴

结局图鉴展示：

- 结局名称
- 结局标题
- 解锁状态
- 达成时间
- 结局简介
- 重新观看入口

未解锁结局显示为：

```text
???
```

---

### 13.4 事件回想

事件回想用于回看已完成剧情节点。

回看规则：

- 不影响当前正式进度
- 不改变变量
- 不覆盖存档
- 可从回想页进入
- 可选择“从此处重新开始”创建新进度

---

### 13.5 LINE 回想

LINE 回想单独分类。

原因：LINE 是本产品的重要关系演出，不应混在普通事件回想里。

LINE 回想内容包括：

- 初期夜聊
- Nagi 主动消息
- 选择洁
- 输给凛后消息
- 异地联系
- 关键战前后消息

---

### 13.6 TRUE END 彩蛋

TRUE END 达成后解锁：

```text
他的求婚
```

入口：

- TRUE END 结局页
- 结局图鉴
- 特殊彩蛋页

彩蛋解锁后可重复观看。

## 15. 视觉风格原则

### 15.1 总体方向

视觉风格应服务于：

```text
日乙感
Nagi 的冷白、低反应、疏离感
玩家的高位、克制、柔软
恋爱表层与世界底层的张力
```

不建议全局使用过度粉玫瑰、花瓣循环、强少女装饰。

建议主色方向：

- 冷白
- 蓝灰
- 深夜蓝
- 低饱和粉
- 玫瑰金点缀
- 半透明玻璃感可保留，但克制使用

---

### 15.2 装饰原则

花瓣、爱心、闪光等元素应作为场景化演出使用。

适用场景：

- 七夕
- 生日
- 夏日祭
- 告白
- TRUE END
- 求婚彩蛋

不建议全局常驻循环。

---

### 15.3 网页版视觉

网页版不重新铺满桌面横屏。

建议：

```text
桌面背景 + 居中手机画布
```

手机画布内保持移动端主体验。

## 17. 验收标准

### 17.1 产品体验验收

一版可用产品应满足：

- 玩家可以完整从开场白进入第一部
- 玩家可以设置姓名和 Nagi 称呼
- 玩家可以阅读剧情并点击推进
- 玩家可以做选择
- 选择后能触发不同回应
- 可以进入章节地图
- 可以自动保存与手动存档
- 可以读取存档
- 可以查看 Backlog
- 可以跳过已读文本
- 可以达成至少一个结局
- 可以在结局图鉴中看到已解锁结局
- 可以回看已读事件

---

### 17.2 日乙体验验收

产品不能只做到“文本能播放”，还必须做到：

- 选项有情绪重量
- 路线不是单纯好感累计
- 数值不破坏沉浸
- 玩家能二周目回收
- 图鉴有收集驱动力
- 已读跳过不折磨玩家
- LINE 模式有独立演出感
- Nagi 的低反应和短句被保留
- 玩家不是女霸总管理 Nagi
- 世界、足球、媒体、资本不是空背景

---

### 17.3 内容口径验收

每个重要节点都应通过以下检查：

```text
1. 这一节点是否仍然服务于 Nagi’s Heart，而不是 Ant’s Heart？
2. 表层是否仍然像乙女恋爱？
3. Nagi 是否仍然像 Nagi？
4. 玩家是否既强又软，而不是女霸总或工具人？
5. 足球 / 世界 / 媒体 / 资本是否有真实压力？
6. 选择是否影响关系质感，而不是只加好感？
7. 结局方向是否服务于 Dream / Stay / Bad 的核心命题？
```

## 19. 当前产品结论

Nagi’s Heart 的新版产品方向应定义为：

```text
移动端优先、可独立适配网页的日乙视觉小说产品。
核心不是养成 Nagi，而是在选择中理解 Nagi，并抵达不同关系结局。
```

产品必须把重心放在：

- 剧情阅读
- 选择重量
- 路线分歧
- 结局达成
- 图鉴回收
- 日乙式二周目体验
- Nagi 的心

而不是把它扩展成养成、抽卡或经营系统。

最终体验目标：

```text
玩家不是把 Nagi 推上世界中心。
玩家是在一次次选择里学会：什么时候靠近，什么时候见证，什么时候放手，什么时候不要把爱变成一条太正确的路。
```
