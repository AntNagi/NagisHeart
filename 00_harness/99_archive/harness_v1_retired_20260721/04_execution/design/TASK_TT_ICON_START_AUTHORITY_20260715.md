# TT 任务：App Icon 与 Start 页权威候选包整理

> 任务编号：`TASK-20260715-002`  
> 任务类型：设计成果收口 / 权威候选整理  
> 优先级：P0  
> 负责人：TT  
> 最终视觉确认：Ant大小姐

## 1. 任务目标

请 TT 对自己已经完成的 App Icon 与 Start 页全部历史版本进行自主复盘和收口，整理出一套可供视觉确认、开发接入、测试验收和后续追溯的权威候选包。

TT 是这组平面成果的原设计者，本任务由 TT 自主完成版本比较、最终候选选择和设计说明。PM 一一不预设最终采用 v15、v23 或任何其他版本，也不替 TT 决定底图、构图、字体、颜色和切图方式。

## 2. 开始前先做版本盘点

1. 检查 TT 自己的历史任务、会话产出、本地输出和仓库现有文件。
2. 盘点范围必须覆盖到 TT 实际完成的最新版本，包括已经做到的 v23；不得只看仓库当前散落的 v15 包就直接宣布定稿。
3. 建立简洁版本索引，至少写明：版本号、文件位置、主要变化、是否淘汰、是否进入最终候选。
4. 历史过程稿不需要全部复制进权威包，但版本关系必须可追溯。
5. 如果某个历史文件只存在于旧会话或旧目录，先找回或说明缺失，不得用较旧文件冒充最新成果。

## 3. TT 自主做出的结论

TT 在盘点后自行确定：

- App Icon 最终候选版
- Start 页最终候选版
- 使用的底图和最终构图
- 哪些元素烘焙进成图，哪些元素作为独立切图交给开发
- 为什么选择该版本，为什么淘汰其他关键版本
- Android 与 Web 推荐采用整图实现、分层实现，或两者分别采用不同接法

以上均由 TT 基于自己的设计专业判断整理。一一不提前指定答案。

## 4. 必须交付的权威候选包

建议统一放入：

`design/authority/icon_start_tt/`

至少包含以下内容。

### 4.1 设计方案说明

文件建议：`TT_Icon_Start_Authority_Spec_v1_0.md`

必须说明：

- App Icon 的设计概念、识别重点、主体构图、色彩和边缘处理
- Start 页的设计概念、视觉层级、人物与品牌标题关系、START 操作层级
- 最终候选对应的历史版本号
- 与历史关键版本相比保留了什么、淘汰了什么以及原因
- Icon 与 Start 页之间怎样保持同一品牌识别
- 适用平台与不适用场景
- 仍需 Ant大小姐确认的视觉点

### 4.2 预览效果图

至少提供：

- Icon 单图预览
- Icon 在 Android 常见圆形、圆角矩形等遮罩下的预览
- Icon 在桌面/启动器中的尺寸效果预览
- Start 页 `1080 x 1920` 完整效果图
- Start 页手机画框或真实屏幕比例预览
- 一张 Icon + Start 页总览联系表

预览图只用于验收展示，不得替代原始底图和正式切图。

### 4.3 底图

必须提供 Start 页最终使用的干净底图：

- 不包含品牌标题、START、说明线或其他前景 UI
- 保留完整分辨率
- 写明原始来源、原始尺寸、裁切方式和最终输出尺寸
- 如使用了调色、修图、扩图或局部处理，写明处理内容

### 4.4 关键切图

由 TT 根据最终实现方案自行判断切图范围，至少覆盖实际需要独立控制的元素，例如：

- 品牌标题层
- START 操作层
- 必要的装饰线、标记或光效层
- App Icon 的 1024 主文件、商店文件和各端需要的导出文件

所有切图必须：

- 使用透明背景时保持真实 alpha
- 不夹带错误底色、遮罩块或多余画布内容
- 文件名可读且能区分用途
- 写明像素尺寸、在基准画布中的原点位置和锚点

### 4.5 实现规范

必须把设计转成开发可执行数据，不能只写“按效果图还原”。至少包括：

#### Start 页

- 基准画布尺寸和宽高比
- 背景缩放方式：cover / contain / 固定裁切等
- 背景焦点和不同屏幕比例下的裁切保护规则
- 每个前景层的 `x / y / width / height`
- 同时给出基准像素值和相对画布百分比
- 对齐方式、锚点、层级顺序和透明度
- 顶部/底部安全区与不可遮挡区域
- START 的点击热区尺寸，不得只按文字可见像素计算
- 如有动效：时长、alpha、scale、循环方式、停止条件
- 首次进入和后续进入的页面职责保持一致：Start 页只负责进入，不承载主页菜单
- Android Compose 和 Web/CSS 各自推荐实现方式

#### App Icon

- 1024 主文件与商店导出尺寸
- Android `mdpi / hdpi / xhdpi / xxhdpi / xxxhdpi` 对应文件和尺寸
- 是否采用 Adaptive Icon；如采用，提供 foreground / background 及安全区
- 各类系统遮罩下必须保留的视觉安全范围
- 是否允许平台二次圆角、阴影或缩放，以及禁止处理项
- Android 资源目录、资源命名与 Manifest 引用建议
- Web/PWA 使用时的导出尺寸和引用建议（如适用）

## 5. 文件清单与来源追溯

权威候选包内必须有 `MANIFEST.md`，逐项列出：

- 文件名
- 用途
- 尺寸
- 是否为最终候选
- 来源历史版本
- 是否允许开发直接使用
- 被哪个实现位置引用

不得把来源不明、版本不明或只是临时预览的图片放进“开发可直接使用”清单。

## 6. 禁止事项

1. 不要因为仓库当前只集中保存了某个旧版本，就默认它是最终版。
2. 不要由 PM 旧说明、XoXo 旧交付包或 yiyi 当前实现替 TT 决定最终视觉。
3. 不要覆盖历史源文件。
4. 不要把预览图当底图，把带文字整图当干净底图。
5. 不要只交 PNG 而没有实现尺寸、位置和接入方法。
6. 不要直接把权威候选宣布为正式权威；最终仍需 Ant大小姐确认。

## 7. 自检与回报

提交前至少检查两遍：

### 第一遍：版本与文件完整性

- 是否盘点至实际最新版本 v23
- 最终候选是否来自 TT 自己确认的正确版本
- 设计说明、预览、底图、切图、实现规范和 MANIFEST 是否齐全
- 是否存在丢图、错图、重复图或历史版本冒充最终版

### 第二遍：开发可执行性

- 开发能否不猜测地得到每层位置、尺寸、锚点、层级和适配方式
- Android Icon 尺寸、资源路径和安全区是否明确
- Start 页在不同屏幕比例下是否有裁切规则
- 预览效果与切图组合是否一致

完成后：

1. 将 `TASK-20260715-002` 状态改为 `review`。
2. 提交权威候选包和版本索引。
3. 回报 PM 一一：最终候选版本、输出路径、待 Ant大小姐确认项。
