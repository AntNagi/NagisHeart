## 4. P0 解析识别补丁 - 玩家对白误判为旁白

第一章现网表现已证明：有一批 `{{playerName}}:` 开头的玩家对白，被运行时错误渲染成了旁白样式。

这不是设计问题，也不是 script 文案本身写成了旁白，而是开发侧识别规则不够硬。请直接按以下解析规则修正：

### 4.1 解析硬规则

- 任何存在合法 `speaker` 的台词行，一律视为人物对白。
- `{{playerName}}` 是合法 `speaker`，与 `Nagi`、`绘心`、`帝襟杏里`、`JFA会长`、`一一` 同级。
- 只有无 `speaker` 行，才允许进入短旁白 / 长旁白 / 连续叙事块判定。
- 有 `speaker` 的短句，例如“他是谁？”“不用。”，不得因为字数短、语气轻、前后夹着旁白，或处于投资会议场景，就降级成旁白。
- 连续叙事块合并边界中，只要出现任意 `speaker` 行，必须强制断开当前旁白块。

### 4.2 本轮必须覆盖的错误样例

以下句子在 script 中都是 `{{playerName}}:` 开头，必须渲染为玩家对白，不得进入旁白容器：

- `{{playerName}}: 他是谁？`
- `{{playerName}}: 不用。`
- `{{playerName}}: 框架没有问题。第一阶段追加投资可以签。`
- `{{playerName}}: 我投资的是蓝色监狱的系统能力。`
- `{{playerName}}: 如果后续只做包装，不做竞技兑现，这笔投资会重新评估。`

### 4.3 实现要求

- 不要再依赖文案长度、语气、斜体样式或前后文去猜测是否旁白。
- 先按 `speaker` / 无 `speaker` 做一级分流，再做旁白细分。
- 若当前 story-data 仍只有 `speaker`、`text`、`display`，也必须先基于 `speaker` 兜住对白识别，不允许继续误判。
- 下一版开发回信中，必须明确写出你在哪个解析 / 映射 / 渲染层修了这条规则。
