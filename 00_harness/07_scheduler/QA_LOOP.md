# 测试循环

## 目标

验证某个已声明完成的实现或内容变更，并留下明确的通过、失败或阻塞结果，供 PM 和下一步执行使用。

## 触发时机

出现以下任一情况时，执行一次测试循环：

- 执行者声明某个修复或交付内容可供验证
- PM 要求抽查或回归验证
- 某个班次需要确认当前风险状态

## 必读输入

按以下顺序读取：

1. `..\00_project\PM_PROJECT_CONTEXT_BRIEF.md`
2. `..\00_project\WORKSPACE_STRUCTURE.md`
3. 相关规划记录
4. 相关 PM 分派或验收说明
5. 相关执行输出 / 声称完成的结果
6. 完成以上之后，再读取或运行最小必要范围的项目文件

## 步骤

1. 确认当前要验证的对象是什么。
2. 根据任务和决策上下文，确认预期结果是什么。
3. 执行最小但有意义的验证，例如：
   - 定向功能验证
   - 回归检查
   - UI / 状态确认
   - 数据 / 路径验证
4. 将结果记录为以下之一：
   - 通过
   - 失败
   - 阻塞
5. 如果失败，记录：
   - 具体症状
   - 出现位置
   - 是否为回归问题
6. 如果阻塞，记录：
   - 是什么阻止了验证
   - 缺少什么输入或环境
7. 如果通过，记录：
   - 实际验证了什么
   - 还有哪些残余风险或未测边缘

## 产出

测试产出必须包含：

- 验证目标
- 验证方法
- 验证结果
- 证据摘要
- 下一步建议

## 停止条件

当结果已经写得足够清楚，使 PM 不必回看原始聊天也能决定下一步时，结束本轮。

## 不要做

- 不要默认接受不清晰的范围
- 不要在测试阶段擅自改写产品意图
- 不要在没写清楚“实际验证了什么”的情况下标记通过
