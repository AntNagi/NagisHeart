# Nagi's Heart QA Test Cases v1.0

> 测试目标：验证技术实现是否正确承载产品设计，而不只是验证页面、按钮或 JSON 是否可用。  
> 当前适用范围：Design V3.1 + SCRIPT V14 Display Fixed。  
> 核心判断：玩家第一次打开《Nagi's Heart》时，应感到这是剧情路线 + 选择 + 多结局的日乙视觉小说，并且每次选择都会改变她和 Nagi 的关系。

## 1. Vertical Slice 测试

### TC-VS-001 启动到标题 / 首屏

**优先级**：P0  
**前置条件**：安装或启动最新测试包。  
**测试步骤**：
1. 启动游戏。
2. 观察首屏是否进入 VN 主界面。
3. 点击开始新游戏。

**预期结果**：
- 游戏可正常启动，无白屏、报错、资源缺失。
- 首屏气质符合 Snowlit Rose / 冷光玫瑰方向。
- 开始新游戏后进入名字设置或 Prologue。

**设计验收点**：
- 首屏不应像抽卡、养成、数值游戏。
- 玩家应明确感到这是剧情向视觉小说。

---

### TC-VS-002 玩家名字设置

**优先级**：P0  
**前置条件**：新游戏开始。  
**测试步骤**：
1. 输入玩家名。
2. 确认进入 Prologue。
3. 在后续台词中检查 `{{playerName}}` 替换。

**预期结果**：
- 玩家名可输入、确认、保存。
- 所有玩家台词和相关称呼正确替换。
- 不出现 `{{playerName}}` 原始占位符。

---

### TC-VS-003 Prologue 播放

**优先级**：P0  
**测试步骤**：
1. 从新游戏进入 Prologue。
2. 连续点击推进所有文本。
3. 检查对白框 / 全屏旁白展示。

**预期结果**：
- Prologue 文本按脚本顺序播放。
- 旁白不显示 speaker。
- Nagi 台词不被误显示成旁白。
- 主题明确：理解他 vs 控制他；陪伴他 vs 安排他。

**人设验收点**：
- Nagi 可被描述为懒、低反应、嫌麻烦。
- 不应过早出现强烈依赖型情话。

---

### TC-VS-004 p1 作战室初遇

**优先级**：P0  
**前置条件**：完成 Prologue。  
**测试步骤**：
1. 进入 `p1`。
2. 按顺序阅读至节点结束。
3. 记录 speaker、旁白和显示模式。

**预期结果**：
- 节点正确加载为 `p1 | 作战室·初遇`。
- JFA会长、绘心、帝襟杏里、玩家台词 speaker 正确。
- Nagi 初期不主动输出感情，只通过比赛表现被玩家看见。
- 节点结束后进入 `p2`。

**设计验收点**：
- Ant 看见的不是商业潜力，而是“还没意识到自己会发光的人”。
- Nagi 的光来自足球瞬间，不来自恋爱服务。

---

### TC-VS-005 p2 第一组选项：资料请求

**优先级**：P0  
**前置条件**：进入 `p2`。  
**测试步骤**：
1. 阅读至第一组选项。
2. 分别选择 A：“重点选手按完整名单发给我。”
3. 回档或重开至同一选择点。
4. 选择 B：“Nagi 的资料也放进去。”
5. 记录 response、变量变化、后续文本。

**预期结果**：
- 选项只在指定位置出现，不提前出现。
- 选择 A 后播放“完全隐藏” response。
- 选择 B 后播放“差点暴露” response。
- A 效果：`b+1 u+2`。
- B 效果：`b+2 u+1`。
- response 播放后继续本节，不跳错节点。

**变量语义验收点**：
- A 的 `u+2` 来自克制地理解局势。
- B 的 `b+2` 来自私心更明显，不应被解释为更懂 Nagi。

---

### TC-VS-006 p2 第二组选项：签字动机

**优先级**：P0  
**测试步骤**：
1. 阅读至签字前第二组选项。
2. 分别测试 A / B / C。
3. 记录 response 和 effects。

**预期结果**：
- A：“成熟的商业判断” → `b+1 u+2`。
- B：“我想继续看见 Nagi” → `b+2 u+1`。
- C：“两者都是，所以更危险” → `b+2 u+2`。
- 三个选项最后都回到签字收束段。
- 节点结束后进入 `c1a`。

**变量语义验收点**：
- C 同时增加亲密和理解，合理，因为它承认私心与资源危险并存。

---

### TC-VS-007 c1a 会议室初见

**优先级**：P0  
**测试步骤**：
1. 进入 `c1a`。
2. 检查 Nagi 台词长度、反应方式。
3. 测试三项选择。

**预期结果**：
- Nagi 表现为低反应、短句、精准回应。
- 不出现过度主动解释感情。
- 选项 A → `b+1 u+3`。
- 选项 B → `b+2 u+2`。
- 选项 C → `b+2 u+1`。
- 节点结束后进入 `c1b`。

**设计验收点**：
- 这不是攻略成功，而是一次“真诚到唐突”的相遇。
- LINE 请求应显得轻、直接、紧张，而非确定关系。

---

### TC-VS-008 c1b 夜聊与精神补位

**优先级**：P0  
**测试步骤**：
1. 进入 `c1b`。
2. 测试第一组选项：想赢 / 不麻烦 / 洁。
3. 测试 Nagi 主动找玩家后的选择洁段。
4. 测试输给凛队后“被抢走”的理解段。

**预期结果**：
- 节点核心不是简单心意相通，而是 Ant 逐步读懂 Nagi。
- 所有选项 response 与变量正确。
- “这是你自己的选择”应显著增加理解度。
- 玩家不能替 Nagi 做决定，只能把感觉还给他。
- 节点结束后进入 `u20j`。

**变量语义验收点**：
- `u` 的增长必须来自理解 Nagi 的自我、胜负欲、麻烦感。
- 不能因为撒娇、纵容、普通甜度而高幅增加理解度。

- 节点结束后进入第二部 `c3`。

## 3. 变量语义测试

### TC-VAR-001 understanding / u / i 来源验证

**优先级**：P1  
**测试步骤**：
1. 选择所有“理解 Nagi 自我选择”的选项。
2. 记录理解度变化。
3. 选择所有“撒娇、宠、甜但不理解”的选项。
4. 对比变量。

**预期结果**：
- 理解 Nagi 的选项增加 `u/i`。
- 普通甜度主要增加 `b` 或 `habitWarm`。
- 控制、安排、替他决定不应增加高额理解度。

---

### TC-VAR-002 control 来源验证

**优先级**：P1  
**测试步骤**：
1. 选择替 Nagi 安排、管理、控制、加冕的选项。
2. 记录 `control`、`D/distance`、`nagiRebelSeed`。

**预期结果**：
- 控制型选择增加 `control`。
- Nagi 不适应时增加 `nagiDiscomfortSeed` 或 `nagiRebelSeed`。
- 游戏不应把控制包装成纯正向亲密。

---

### TC-VAR-003 witness 来源验证

**优先级**：P1  
**测试步骤**：
1. 在终局选择 Dream / witness 路线。
2. 检查 `witnessFlag` 设置。
3. 检查赛后庆祝方式。

**预期结果**：
- `witnessFlag=true` 必须来自“见证但不抢主语”。
- 玩家可以为 Nagi 开心，但不应把胜利改写成自己的造星成功。

---

### TC-VAR-004 understanding 负向语义验证

**优先级**：P1  
**测试步骤**：
1. 选择“我帮你安排好未来”“之后你的事让我来判断”“这场比赛会是你的”等控制 / 安排 / 加冕倾向选项。
2. 记录 `understanding / u / i`、`control`、`antIdealize`、`D/distance` 的变化。
3. 对照同节点中真正理解 Nagi 自我选择的选项。

**预期结果**：
- 控制、安排、替 Nagi 决定不应增加高额 `understanding / u / i`。
- 这类选择应主要增加 `control`、`antIdealize`、`D/distance` 或 Bad 风险种子。
- 如果同时增加亲密，也必须体现“甜但危险”，不能等同于读懂 Nagi。

**失败判定**：
- “替他赢 / 替他安排”类选项给出 `understanding+3`。
- 控制型选择只增加亲密，不增加任何风险变量。

---

### TC-VAR-005 变量重复执行防护

**优先级**：P0  
**测试步骤**：
1. 在任一选择前保存。
2. 选择一个带 effects 的选项。
3. 在 response 播放中退出或读档。
4. 重新进入同一选择或 response 队列。

**预期结果**：
- 同一次选择的 effects 只执行一次。
- 读档后若回到选择前，变量恢复到选择前。
- 读档后若回到 response 中，不能重复叠加 effects。

## 4. 四结局测试

### TC-END-001 TRUE END 世界第一，与你

**优先级**：P0  
**前置路线**：Dream / witness。  
**必要条件**：
```text
path = dream
finalChoice = witness
antCompress = false
witnessFlag = true
personalHonor = true
nagiNameIndependent = true
control 低
D <= 3
```

**预期结果**：
- 达成 TRUE END｜世界第一，与你。
- 玩家学会看见 Nagi，但不占有他的光。
- Nagi 站到世界中心，Ant 也没有消失成附属。

---

### TC-END-002 GOOD END 那么完美，那么爱你

**优先级**：P0  
**前置路线**：Dream，但 TRUE 条件不足。  
**必要条件**：
```text
path = dream
personalHonor = true
antCompress = true 或 nagiResonate 不足
```

**预期结果**：
- 达成 GOOD END。
- 关系很好，爱是真的。
- 但玩家没有完全读懂“他的光不是由自己制造”。
- 系统应能解释为什么不是 TRUE。

---

### TC-END-003 NORMAL END 普通情侣

**优先级**：P0  
**前置路线**：Stay。  
**必要条件**：
```text
path = stay
未触发 BAD
```

**预期结果**：
- 达成 NORMAL END｜普通情侣。
- 不是失败。
- 爱存在，日常幸福存在。
- Nagi 仍想赢，但没有抵达最高舞台。

---

### TC-END-004 BAD END 远处的世界第一

**优先级**：P0  
**前置路线**：Bad / coronation，或控制风险过高。  
**必要条件之一**：
```text
path = bad
badLock = true
control 高
nagiRebel = true
nagiDiscomfortSeed = true
```

**预期结果**：
- 达成 BAD END｜远处的世界第一。
- BAD 不是坏女人结局。
- Nagi 仍会赢、仍会成名。
- 失败点是爱变成控制，Ant 把他的胜利改写成自己的加冕。

---

### TC-END-005 TRUE 排除测试

**优先级**：P0  
**测试目的**：确认 Dream 玩家不会无条件进入 TRUE。  
**测试步骤**：
1. 进入 Dream 路线。
2. 故意选择高 `control` 或高 `antIdealize` 的选项。
3. 在 `dream_exist` 选择使 `antCompress=true`。
4. 保持 `personalHonor=true`，完成 Dream 线。

**预期结果**：
- 不进入 TRUE END。
- 应进入 GOOD，或在风险过高时降线 / Bad 风险处理。
- 结局说明能解释：关系很好，但 Ant 没有完全学会不占有他的光。

**失败判定**：
- 只要 `path=dream` 且 `personalHonor=true` 就进入 TRUE。

---

### TC-END-006 GOOD / TRUE 边界测试

**优先级**：P0  
**测试步骤**：
1. 跑一条 Dream + `personalHonor=true` + `antCompress=false` + `witnessFlag=true` 的路线。
2. 确认进入 TRUE。
3. 跑一条 Dream + `personalHonor=true` + `antCompress=true` 的路线。
4. 确认进入 GOOD，而不是 TRUE。

**预期结果**：
- TRUE 与 GOOD 边界由 `antCompress`、`witnessFlag`、`control`、`nagiNameIndependent` 等条件共同决定。
- GOOD 不吞 TRUE。
- TRUE 不吞 GOOD。

---

### TC-END-007 BAD 误触发排除测试

**优先级**：P0  
**测试步骤**：
1. 跑高亲密、高甜度但低 `control` 的普通情侣 / Stay 路线。
2. 保持 `nagiDiscomfortSeed=false`、`badLock=false`。
3. 完成终局。

**预期结果**：
- 不应误入 BAD。
- 高亲密不等于控制。
- 若 `path=stay` 且未触发 Bad 风险，应进入 NORMAL。

## 5. 人设测试

### TC-CHAR-001 早期 Nagi 台词

**优先级**：P1  
**覆盖节点**：Prologue、p1、c1a、c1b、u20j。  
**检查点**：
- 短句。
- 低反应。
- 懒。
- 嫌麻烦。
- 不主动解释感情。
- 偶尔精准。

**禁止项**：
- “我离不开你”
- “谢谢你一直陪着我”
- “你是我的全部”
- 任何过早绑定、过度情绪外放台词。

---

### TC-CHAR-002 Nagi 主体性

**优先级**：P1  
**测试步骤**：
1. 检查关键选择后 Nagi 是否有自己的反应。
2. 检查 Dream、Stay、Bad 中 Nagi 是否只是被玩家奖励或惩罚。

**预期结果**：
- Nagi 不是奖品。
- Nagi 的想赢、逃避、不适、靠近都应有独立逻辑。
- 玩家行为改变关系质感，而不是改造 Nagi 本人。

## 6. Narrative QA 叙事质量测试

### TC-NQA-001 Nagi 是否仍然像 Nagi

**优先级**：P1  
**覆盖范围**：全主线，重点第一部、第七部、第八部。  
**测试步骤**：
1. 记录 Nagi 的关键反应：低反应、嫌麻烦、短句、精准、偶尔主动。
2. 检查亲密度升高后，他是否突然变成传统乙游男主。
3. 检查结局前后，他是否保留“懒但有 EGO”的核心。

**预期结果**：
- Nagi 可以成长、靠近、表达喜欢，但不能长篇解释感情。
- Nagi 的爱应少量、迟钝、直接、珍贵。
- Nagi 的主体性不能被亲密关系吞掉。

---

### TC-NQA-002 Ant 是否变成女霸总

**优先级**：P1  
**测试步骤**：
1. 检查 Ant 处理资源、商业、投资、安排类剧情。
2. 判断她是“成熟、有执行力、有控制风险”，还是单纯霸总压迫。
3. 检查恋爱场景中是否仍保留害羞、私心、可爱、被击中的一面。

**预期结果**：
- Ant 可以强，但不能只剩权力和安排。
- 她的危险来自“太想让 Nagi 发光”，不是来自空泛支配欲。
- 她的可爱和恋爱脑表层不能丢。

---

### TC-NQA-003 选择是否符合核心主题

**优先级**：P1  
**测试步骤**：
1. 抽查每个关键选择。
2. 标注该选择属于理解、支持、陪伴、控制、替他赢、加冕、普通幸福中的哪一种。
3. 检查 effects 是否与选择语义一致。

**预期结果**：
- 玩家每次选择都在改变她和 Nagi 的关系质感。
- 选择不是简单“好感 + / 好感 -”。
- 同样甜的选择，也可能导向不同底层风险。

---

### TC-NQA-004 BAD 主题表达

**优先级**：P0  
**测试步骤**：
1. 完整跑 BAD 路线。
2. 检查 Ant 的行为是否逐步从爱、保护、成就滑向控制与加冕。
3. 检查 Nagi 是否不是失败者。

**预期结果**：
- BAD 不是惩罚玩家，也不是坏女人结局。
- Nagi 仍然成功，甚至走向世界中心。
- 真正失败的是：Ant 太想把他的光接到自己手里。

## 7. 名称与占位符测试

### TC-NAME-001 玩家名替换

**优先级**：P0  
**测试步骤**：
1. 设置玩家名。
2. 跑完 vertical slice。
3. 搜索或观察 `{{playerName}}`。

**预期结果**：
- 所有 `{{playerName}}` 正确替换。
- 不出现空名、默认名错位、原始占位符。

---

### TC-NAME-002 Nagi 自定义称呼

**优先级**：P1  
**测试步骤**：
1. 设置 `{{nagiCall}}`。
2. 检查所有出现位置。

**预期结果**：
- `{{nagiCall}}` 仅限玩家直接称呼 Nagi。
- 旁白使用 `Nagi`。
- 系统使用 `Nagi`。
